# 🔍 CI/CD 구축이 기존 코드에 미치는 영향

**"기존 코드는 전혀 변경되지 않습니다!"**

---

## ✅ 핵심 답변

### ❌ 변경되지 않는 것들
- ✅ **기존 코드 파일** (`.ts`, `.tsx`, `.js` 등)
- ✅ **비즈니스 로직**
- ✅ **API 엔드포인트**
- ✅ **데이터베이스 스키마**
- ✅ **프론트엔드 컴포넌트**
- ✅ **기존 기능들**

### ✅ 새로 추가되는 것들
- 📁 `.github/workflows/` 폴더 (새로 생성)
- 📄 `deploy-backend.yml` 파일 (새로 생성)
- 📄 `deploy-frontend.yml` 파일 (새로 생성)
- 🔧 GitHub Secrets 설정 (환경 변수만)

---

## 📊 상세 분석

### 1. 기존 코드 변경 없음

**예시: 기존 파일들**
```
backend/src/
  ├── auth/          ← 변경 없음 ✅
  ├── student/       ← 변경 없음 ✅
  ├── ai/            ← 변경 없음 ✅
  └── ...

frontend/src/
  ├── app/           ← 변경 없음 ✅
  ├── components/    ← 변경 없음 ✅
  └── ...
```

**CI/CD는 코드를 실행하는 방식만 바꿀 뿐, 코드 자체는 건드리지 않습니다.**

---

### 2. 새로 추가되는 파일들

**추가될 파일 구조:**
```
프로젝트 루트/
  └── .github/                    ← 새 폴더
      └── workflows/              ← 새 폴더
          ├── deploy-backend.yml  ← 새 파일
          └── deploy-frontend.yml ← 새 파일
```

**이 파일들은:**
- ✅ 기존 코드와 완전히 분리됨
- ✅ 배포 프로세스만 정의함
- ✅ 코드 동작에 영향 없음

---

### 3. 기존 스크립트 활용

**현재 `package.json`에 이미 있는 스크립트:**
```json
{
  "scripts": {
    "test": "jest",           ← CI/CD에서 사용
    "build": "next build",    ← CI/CD에서 사용
    "start": "next start"     ← CI/CD에서 사용
  }
}
```

**CI/CD는 기존 스크립트를 그대로 사용합니다!**
- 새로운 스크립트 추가 불필요
- 기존 스크립트 이름 변경 불필요

## ⚠️ 주의사항 (긍정적 영향)

### 1. 테스트가 실패하면 배포 안 됨

**Before CI/CD:**
```
코드 작성 → 테스트 안 돌려도 배포 가능 ❌
→ 버그가 있는 코드도 배포됨
```

**After CI/CD:**
```
코드 작성 → 테스트 자동 실행
→ 테스트 실패 시 배포 중단 ✅
→ 버그가 있는 코드는 배포 안 됨
```

**영향:** 더 안전한 배포! (긍정적)

---

### 2. 빌드 실패 시 배포 안 됨

**Before CI/CD:**
```
코드 작성 → 빌드 에러 있어도 수동 배포 가능 ❌
→ 에러 있는 코드도 배포됨
```

**After CI/CD:**
```
코드 작성 → 빌드 자동 실행
→ 빌드 실패 시 배포 중단 ✅
→ 에러 있는 코드는 배포 안 됨
```

**영향:** 더 안정적인 배포! (긍정적)

---

### 3. 린터 에러 시 배포 안 됨 (선택사항)

**CI/CD에 린터 체크 추가 시:**
```
코드 작성 → 린터 자동 실행
→ 린터 에러 시 배포 중단 ✅
→ 코드 품질이 낮은 코드는 배포 안 됨
```

**영향:** 더 깔끔한 코드! (긍정적)

## 🔄 개발 워크플로우 변화

### Before CI/CD

```
1. 코드 작성
2. 로컬 테스트 (선택사항)
3. GitHub 푸시
4. 수동으로 서버 접속
5. 수동으로 빌드
6. 수동으로 배포
```

**문제점:**
- 테스트 안 해도 배포 가능 ❌
- 빌드 에러 있어도 배포 가능 ❌
- 실수 가능성 높음 ❌

### After CI/CD

```
1. 코드 작성
2. GitHub 푸시
   ↓ (자동)
3. 테스트 자동 실행 (필수)
4. 빌드 자동 실행 (필수)
5. 배포 자동 실행 (필수)
```

**장점:**
- 테스트 통과해야만 배포 ✅
- 빌드 성공해야만 배포 ✅
- 실수 방지 ✅

- 일관성 있음

**결과:** 더 나은 협업! 👥
